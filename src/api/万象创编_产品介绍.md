# 万象创编 — 产品介绍与实现纲要

**定位**：你的个人 **构思资产库** + **创编工作台**。把分散在备忘录、浏览器收藏、仓库里的「技能片段、规范写法、剧本脑洞、产品页面/接口设想、网络工作流」**结构化落库**；再基于已存内容，由 AI **归纳、联想、扩写、生成新工作流与新构思**，形成「越存越会用」的闭环。

**不做**：替你做最终代码合并进业务仓库（仍以你本地 Git 与审阅为准）；**不替代**版权清晰的商用剧本外包流程，仅作个人创作辅助。

---

## 1. 产品怎么设计，能帮助存储构思与创造

### 1.1 核心目标

- **统一容器**：下列各类素材用**同一套「构思条目」模型**承载，区别主要在 **类型、标签、正文形态**（Markdown / 纯文本 / 外链快照），避免多个互不相通的笔记应用。
- **可检索、可关联**：支持标签、全文检索、类型筛选；条目之间可标记 **衍生关系**（「由 A 启发得到 B」），便于追溯思路链。
- **可喂给 AI**：存库内容经授权范围（默认仅本人数据）作为 **RAG 或上下文片段** 进入「创编」能力，实现你提出的 **「根据我存的构思和文件，分析和创造更多构思与工作流」**。
- **版本与来源**：重要长文（如 `.cursorrules` 草案、剧本大纲）支持 **版本历史**；从网络收藏的工作流保留 **原始 URL 与抓取时间**，避免日后链接失效时全无依据。

### 1.2 与各类需求一一对应

| 需求 | 产品能力 |
|------|----------|
| 前端 / 后端 / 算法 / 产品 / UI 等各类 **SKILLS**，以及 **CLAUDE.md、cursorrules、harness**（如 `.cursor/harness` 或 Agent 测试脚手架说明）等规范写法 | 类型：`SKILL` 或 `RULE_DOC`（可用子类型区分 skill / 规则文档 / harness）；正文存 Markdown；标签区分领域（`frontend`、`backend`、`algo` 等）并支持 `claude`、`cursor`；支持 **版本对比**；可附 **文件引用**（见下表）。 |
| **小说、影视剧本构思** | 类型：`SCRIPT`；字段可扩展：**体裁**（长篇/短篇/剧集）、**梗概**与**分场**分文档或同一文档分节。 |
| **历史产品页面与接口构思** | 类型：`PRODUCT_NOTE`；子字段：**路由/页面名**、**接口路径与方法**、**请求响应要点**（结构化 JSON 或 Markdown 表格）。 |
| **网络上各类构思与工作流** | 类型：`WORKFLOW` 或 `CLIP`；**来源 URL**、**摘要**、**原文快照**（可选）；标签如 `web`、`workflow`。 |
| **AI 分析并创造更多构思与工作流** | 独立 **「AI 创编」** 能力：选择条目/标签范围 → 选择任务（扩写、脑暴、反推工作流、合并重复）→ 输出 **新草稿条目**（默认未发布，你可再编辑入库）。 |

### 1.3 产品原则

- **你的数据默认私有**；若未来多设备同步，需明确加密与访问控制策略。
- **AI 生成内容**均带有 **来源引用**（引用了库中哪些条目）与 **生成时间**，便于审计与回滚。
- **长文编辑**建议与 Markdown 编辑器体验一致（分屏预览可选）。

## 2. 前端页面怎么设计

### 2.1 页签（与 `chat/page.tsx` 一致的模式）

与现有「晤言」一致，使用 **`role="tablist"`**、`role="tab"`、`role="tabpanel"`，并配套 **`aria-selected`、`aria-controls`、`tabIndex`** 等，保证键盘与读屏可用。

建议 **`const TABS` 常量**（文案可按迭代微调），例如：

```text
const TABS = ["构思总览", "技能与规范", "剧本与叙事", "产品与接口", "工作流收藏", "AI 创编"] as const
```

| Tab | 职责简述 |
|-----|----------|
| **构思总览** | 最近更新、按标签/类型筛选的卡片或表格入口；支持搜索框；快捷新建。 |
| **技能与规范** | 筛选类型为 SKILL / RULE_DOC；列表 + 详情抽屉或侧栏；强调 Markdown 与附件。 |
| **剧本与叙事** | 类型 `SCRIPT`；可按体裁、状态（草稿/归档）筛选；长文阅读友好排版。 |
| **产品与接口** | 类型 `PRODUCT_NOTE`；表格列突出 **页面/接口** 关键词；可嵌入简易 API 表格模板。 |
| **工作流收藏** | 类型 `WORKFLOW`/`CLIP`；突出 **来源链接** 与摘要卡片；外链新开 `rel="noopener noreferrer"`。 |
| **AI 创编** | 选择范围（标签/时间/类型）→ 输入任务说明 → 展示生成结果 → **一键存为新构思**（进入草稿）。 |

若希望 **减少 Tab 数量**，也可合并为 **「素材库」单 Tab + 顶栏类型筛选**，但会弱化「按心智模型分区」；上表为 **与上文需求对齐的推荐拆法**。

### 2.2 通用交互

- **列表**：加载中、空态、分页（条目多时）；搜索与多条件筛选。
- **详情**：标题、类型、标签、正文（Markdown 渲染）、附件列表、关联条目、版本历史入口。
- **新建/编辑**：表单或分步表单；**标签**支持多选与创建；**类型**为枚举单选。
- **无障碍**：表单控件带 `label` 或 `aria-label`；图标按钮需 `aria-label`；异步结果区域 `aria-live="polite"`。

## 3. 表怎么设计 — 需要哪些表

表名可加统一前缀（如 `wx_` 表示万象创编），以下为逻辑模型。

### 3.1 核心：构思条目

| 表名（建议） | 用途 |
|--------------|------|
| `idea_entry` | 主表：`id`，`title`，`type`（枚举：SKILL、RULE_DOC、SCRIPT、PRODUCT_NOTE、WORKFLOW、CLIP 等），`body`（Markdown 或长文本），`status`（草稿/已发布/归档），`source_url`（可选），`source_snapshot`（可选，外链快照文本），`created_at`，`updated_at`，`owner_id`（若多用户）。 |
| `idea_tag` | 标签名（唯一或归一化）。 |
| `idea_tag_rel` | 多对多：`idea_id`，`tag_id`。 |
| `idea_version` | 版本历史：`idea_id`，`version_no`，`body`，`edited_at`，`editor_id`（可选）。 |

### 3.2 附件与关联

| 表名（建议） | 用途 |
|--------------|------|
| `idea_attachment` | 附件：`idea_id`，`storage_key` / URL，`filename`，`mime`，`size`；对应上传至对象存储或本地。 |
| `idea_relation` | 条目间关系：`from_id`，`to_id`，`relation_type`（`DERIVED_FROM`、`RELATED`、`DUPLICATE_OF` 等）。 |

### 3.3 产品与接口（可选结构化扩展）

若希望 **强结构化查询**（例如「列出所有 POST /api/foo」），可增加：

| 表名（建议） | 用途 |
|--------------|------|
| `product_api_stub` | `idea_id`（关联某条 PRODUCT_NOTE），`method`，`path`，`summary`，`json_schema`（可选）。 |

也可仅用 `idea_entry.body` 存 Markdown 表格，**第一版可不做**该表，避免过度设计。

### 3.4 AI 创编记录

| 表名（建议） | 用途 |
|--------------|------|
| `ai_generation_job` | `id`，`prompt_user`，`scope_tags`（JSON），`scope_idea_ids`（JSON），`result_body`，`referenced_idea_ids`（JSON，引用溯源），`model`，`created_at`，`status`。 |
| `ai_generation_output_draft`（可选） | 与 `idea_entry` 草稿联动，或直接在 job 表存结果后由用户「转正」为 `idea_entry`。 |

## 4. 后端接口需要怎么设计

**约定**：统一前缀如 `/codelab/wanxiang`（与现有 `/codelab` context-path 对齐）；响应结构可与现有 `CommonResponse` 一致。

### 4.1 构思 CRUD

- `GET /ideas` — 分页；查询参数：`type`、`tag`、`q`（标题/正文关键词）、`status`、`sort`。  
- `GET /ideas/{id}` — 详情（含标签、附件列表、关联条目摘要）。  
- `POST /ideas` — 新建。  
- `PUT /ideas/{id}` — 更新（写入时可选生成新版本到 `idea_version`）。  
- `DELETE /ideas/{id}` — 软删除或归档（建议软删）。  

### 4.2 标签

- `GET /tags` — 标签列表（可带使用次数）。  
- `POST /tags`（可选）— 新建标签（若允许前端创建）。  

### 4.3 附件

- `POST /ideas/{id}/attachments` — 上传（multipart 或预签名 URL）。  
- `DELETE /attachments/{attachmentId}` — 删除附件。  

### 4.4 关联

- `POST /ideas/{id}/relations` — 建立 `from`/`to`/`relation_type`。  
- `GET /ideas/{id}/relations` — 关联图列表。  

### 4.5 AI 创编（与第 1.2 节第 6 点对应）

- `POST /ai/expand` — 请求体：`scope`（标签 ID 列表或条目 ID 列表）、`instruction`（用户任务说明）、`output_type`（构思 / 工作流 / 脑暴列表）。返回：`draft_content`、`referenced_ids`、`job_id`。  
- `GET /ai/jobs/{jobId}` — 异步任务时查询状态与结果。  

（若 RAG 在独立向量服务中，可由 **Python 或 Node 侧**完成检索后，将拼接上下文交给大模型；Java 侧负责 **鉴权、落库、计费字段** 等。）

### 4.6 搜索（可选增强）

- `GET /ideas/search` — 若全文检索走 ES/OpenSearch，与 `GET /ideas` 分离或合并为同一接口不同 `mode` 参数。

---

## 5. 小结

**万象创编** = **分类清晰的构思库**（多类素材统一模型）+ **标签、关联、版本** + **附件** + **AI 创编接口**（基于已存内容扩写与工作流生成）。

前端采用与 **`chat/page.tsx` 相同的 `TABS` + `tablist` 页签模式** 组织分区；后端以 **构思主表为中心**，逐步迭代 **版本、关联、AI 任务** 表即可落地。
